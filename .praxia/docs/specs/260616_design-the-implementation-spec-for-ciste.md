# Brainstorm: Design the implementation spec for cisterna's telemetry library (Milestone 1: FastMCP + Cyclopts/Typer observability). Cisterna is a shared Python library giving the ecosystem (bathos, contemplex, xperiri, myxcel, jaxlint, maraxiom, aminx, xtrax) drop-in telemetry for MCP tool calls and CLI invocations. It must be OpenTelemetry-compatible yet preserve the existing JSONL workflows. The spec must be decomposable into a backlog DAG.

## Problem Frame
FRAME.

Desired outcome: a buildable, decomposable implementation spec — modules, public API signatures, event/span schema, exporter abstraction, integration seams, and a migration path — ready to become a backlog DAG.

FIXED constraints (cannot change):
1. Python 3.13. Deps pinned: fastmcp>=3.4, cyclopts>=4.18, opentelemetry-api/sdk.
2. Must NOT break bathos/contemplex existing JSONL archives or their duckdb/jq workflows. JSONL output stays backward-compatible at the field level.
3. Never crash the caller — every telemetry entry point is never-raise, and telemetry stays off the hot path of the traced call.
4. <1ms/call overhead target.
5. Must support BOTH FastMCP v3 (middleware) AND v2 (decorator), and BOTH Cyclopts (the migration target) AND Typer (all 8 current CLIs).
6. arg_keys privacy invariant: parameter NAMES only, never values.
7. Shaped error envelopes returned to transport, never re-raised.

NEGOTIABLE (solution space):
- Whether the core is OTel-SDK-native, OTel-API-only thin, or stdlib-JSONL-first.
- Whether the exporter abstraction reuses OTel SpanProcessor/SpanExporter or a custom pluggable Exporter protocol.
- Whether canonical event names replace, alias, or co-exist with per-server dialects (mcp.call_* vs tool_call_*).
- Integration ergonomics: explicit setup(app) vs zero-config auto-instrument, and whether FastMCP version is auto-detected.
- How deep Cyclopts support goes in M1 vs Typer parity.

SUCCESS CRITERIA: bathos (FastMCP v3 middleware) + contemplex (FastMCP v2 decorator, SYNC) + one Typer CLI can all adopt cisterna with NO loss of current telemetry fidelity; JSONL stays backward-compatible; OTLP export works when configured.

NON-GOALS for M1: the asset-export pillar; rewriting any consumer's business logic.

STEELMAN: distributed tracing and OTLP export come for free; parent/child context propagation across the MCP boundary is solved by OTel's context API rather than hand-rolled; one mental model (spans) for everything; semantic conventions adoptable later with no rework. ## Pre-mortem Record
**User:** Pre-mortem — it's 6 months later and cisterna telemetry has definitively failed. The autopsy:

1. SILENT TELEMETRY DEATH (most likely). The never-raise guarantee worked too well: a misconfigured/raising default exporter (or a dead QueueListener thread — the carried-forward EC-3 gap) swallowed every event, and because nothing ever crashed, no one noticed for weeks. The JSONL files were empty/stale and dashboards quietly flatlined. Root cause: never-raise without a liveness signal. FIX TO BAKE IN: a heartbeat/self-check (periodic "telemetry alive" event + an install-time write probe + a drop counter surfaced via a status call).

2. THE SYNC PATH BIT-ROTTED. All tests and the demo used bathos (v3 async middleware). The contemplex (v2 sync decorator) path had a subtle contextvar/Token bug and silently emitted nothing; contemplex lost telemetry entirely and only found out when debugging an unrelated incident. Root cause: the sync/async fork (R3) was treated as one path with a flag, not two first-class tested paths. FIX: parity test matrix that runs the SAME assertions against BOTH the async middleware and sync decorator wrappers.

3. DROP-ON-FULL ATE THE INTERESTING DATA. Under load the non-blocking exporter dropped events; the busiest tools — exactly the ones worth observing — had the worst coverage, and the drops were invisible. Root cause: silent backpressure. FIX: bounded queue with an explicit, counted, periodically-logged drop metric (never silent).

4. NOBODY ADOPTED IT. install_mcp/install_cli was not actually drop-in: CLIs with an existing @app.callback or chained Typer collided (UM-2), and the per-consumer adapter + shadow-parity ritual was more work than teams would do for "telemetry we already kindof have." cisterna sat at 1 adopter. Root cause: adoption cost > perceived benefit. FIX: make the Typer shim composable with existing callbacks, ship copy-paste adapter recipes for bathos+contemplex, and make shadow-mode a single env flag.

Root cause: the freeze was documented but not enforced. FIX: a conformance test/lint that fails if an M1 adapter emits any non-legacy event name.

6. SCOPE SLIP. Someone pulled OTLP + Cyclopts launcher back into M1 "while we're here," and the milestone slipped two months and shipped half-tested. Root cause: the deferral wasn't load-bearing in the backlog DAG. FIX: encode the deferrals as explicit separate DAG nodes (M1.5/M2) with their own gates, not as TODOs inside M1 nodes.

Net: the failures are all SILENT-degradation and ADOPTION-cost modes, not correctness-of-happy-path. The spec must therefore treat observability-of-the-observer (heartbeat, drop counters, liveness) and adoption ergonomics (composable shims, recipes, one-flag shadow mode) as first-class M1 acceptance criteria, not nice-to-haves.
