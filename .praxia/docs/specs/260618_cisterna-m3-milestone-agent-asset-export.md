# Brainstorm: Cisterna M3 milestone: agent-asset export (native Python) — design for ONE canonical agent asset → MANY surfaces (Claude Code, Cursor, Copilot, Antigravity/Gemini), porting the shape of praxia's Rust export engine to Python

## Problem Frame
Fixed constraints:
- Python 3.13, never-raise convention, py.typed, lazy optional-dep imports
- cisterna.adapters namespace is RESERVED — M3 must not use it
- cyclopts already a core dep; CLI surface must use it
- M3 is the asset-export pillar (not telemetry); telemetry is M1/M2
- The praxia-Rust backend must be slot-in-able later (seam required)
- No breaking changes to existing cisterna.adapters or M2 registry surface

Negotiable:
- Which surfaces to include in M3 vs follow-on milestones
- Whether the asset source is file-driven, live-registry-driven, or both
- Whether to port the abstract capability verb map now or defer
- Whether to port sha256 determinism/attestation now or defer
- Writer abstraction shape (pure emit vs stateful)
- Which subpackages to create (cisterna/assets/, cisterna/export/, cisterna/writers/ are all options)
- CLI command shape (cisterna assets export vs other)

Success criteria: a brainstorm artifact with enough distinct named candidate approaches that the synthesizer can select and spec a concrete M3 implementation plan.

## Idea Pool
- [ai] FORK 1-A — Thin vertical slice: IR + Claude-only + validate gate. Implement AssetBundle IR (5 Emittable types), one ClaudeEmitter (pure emit → {path: str}), and a `cisterna assets validate` subcommand with sha256 drift detection. All other surfaces (Cursor, Copilot, Antigravity) deferred to M3.x. Maximizes depth-of-proof on the IR/emitter contract before proliferating surfaces.
- [ai] FORK 1-B — Full 4-surface parity port in M3. Implement IR + all four emitters (Claude, Cursor, Copilot, Antigravity) simultaneously, matching praxia's full surface spec including PascalCase/camelCase hook event normalization. Highest fidelity to praxia blueprint; highest implementation parallelism risk; testing load is O(surfaces).
- [ai] FORK 1-C — Two-surface slice: Claude + Antigravity (PascalCase hook shape). Both surfaces share PascalCase event names and the same agents/<n>.md path convention; hook aggregation logic is written once and shared. Cursor (camelCase, fail-closed gate) and Copilot (inlined hooks) are structurally different enough to defer without losing hook logic work. Validates hook N:1 aggregation AND the emitter abstraction boundary with two surfaces at lower complexity cost than all four.
- [ai] FORK 1-D — Milestone-split slice: M3 = IR + AssetSource protocol + Claude emitter + CLI scaffold (no validate). M3.1 = Cursor + Copilot emitters. M3.2 = Antigravity + validate + sha256. Trades praxia-parity completeness for faster M3 delivery; leaves the validate gate for a point release where drift detection adds obvious value once users have agent files on disk.
- [ai] FORK 2-A — File-driven asset source (praxia parity). Discover and parse .praxia/manifest.toml + role markdown files on disk. AssetSource reads the manifest, walks asset paths, parses YAML frontmatter, assembles AssetBundle IR. Mirrors praxia's discover→parse→build pipeline exactly. Works even when no cisterna Python process is running. Defers the live-registry question entirely.
- [ai] FORK 2-B — Live-registry-first asset source (cisterna-unique). Use registry._snapshot() to pull ToolEntry objects as the primary asset source. Wrap each ToolEntry in a richer AssetSpec (adds description from __doc__, schema from inspect.signature, optional tags/surface hints). Emits tools the host process has actually registered — no stale manifest drift possible. Cisterna's structural differentiator: praxia can't do this. Requires AssetSpec richer than ToolEntry, but avoids any TOML parsing in the hot path.
- [ai] FORK 2-C — Unified AssetSource protocol: both file-driven and live-registry behind one ABC. Define `AssetSource` as a Protocol with `discover() -> Iterable[AssetSpec]`. Provide two concrete implementations: `ManifestAssetSource` (file-driven, praxia-parity) and `RegistryAssetSource` (live M2 registry). Caller composes them with a `CompositeAssetSource` that merges results. The praxia-Rust backend later becomes a third implementation. Maximum flexibility; maximum surface area for M3.
- [ai] FORK 2-D — Registry-first with file fallback. Primary source is the live M2 ToolEntry registry. When registry is empty or unavailable (e.g., CLI invoked out-of-process), fall back to manifest.toml discovery. No explicit Protocol ABC — just a single `load_assets(registry=None, manifest_path=None) -> list[AssetSpec]` function with branch logic. Simpler than the full Protocol approach; delays the seam formalization until there's a third source to justify it.
- [ai] FORK 3-A — Pure emit model (praxia-identical). Emitter ABC: `emit(bundle: AssetBundle) -> dict[str, str]` where keys are relative paths, values are file contents. No I/O in emitters. A separate thin `write_bundle(output: dict[str, str], root: Path) -> None` function does all filesystem writes. The praxia-Rust backend replaces just the emit() call; the write path stays Python. Easy to hash, snapshot, diff, test without touching the filesystem.
- [ai] FORK 3-B — Stateful Writer class that owns I/O. `Writer(root: Path)` with `write(bundle: AssetBundle) -> WriteSummary`. Encapsulates both emit and filesystem write; WriteSummary records which files were written, their sha256s, and any errors. Makes rollback or dry-run a Writer concern (dry_run=True flag). More OO; harder to test the emit logic in isolation; praxia-Rust backend would replace the whole Writer, not just emit().
- [ai] FORK 3-C — Pure emit + typed WriterSink separation. Two distinct ABCs: `Emitter` (pure, no I/O, returns dict[str,str]) and `WriterSink` (accepts dict[str,str], performs I/O). Default `FileWriterSink` writes to disk. A `MemoryWriterSink` captures output for tests. Future `RustWriterSink` delegates to the praxia-Rust binary. This is the cleanest seam for the "praxia-Rust backend later" requirement because Rust replaces just the Emitter, or just the Sink, independently.
- [ai] FORK 4-A — Port sha256 determinism + drift golden tests + validate subcommand in M3. `cisterna assets validate` computes surface_bundle_sha256() over the sorted {path: contents} map and compares against a stored golden hash. Drift detection is cheap to implement once the pure emit model exists (it's just hash(emit_output)). Golden tests lock the emitter contract and prevent silent regressions. The validate subcommand is immediately useful to plugin authors checking whether their assets are stale.
- [ai] FORK 4-B — Defer sha256/validate to M3.1. Implement M3 without any determinism infrastructure. Add `validate` as a follow-on once users have agent files on disk and there's observable drift to detect. Avoids building attestation infrastructure before anyone has files to attest. Risk: the pure emit model makes hashing trivial to add later, so the cost of deferral is low; but shipping without validate means the first users get no drift detection at all.
- [ai] FORK 4-C — Lightweight content hash without full validate subcommand. Compute sha256 of emitted content and embed it as a provenance comment in each generated agent file (matching praxia's hash-stamp behavior) but provide no `validate` CLI. This gives provenance traceability and idempotency checking without a full drift-detection workflow. The full `validate` subcommand becomes obviously useful once users start editing generated files.
- [ai] FORK 5-A — Port abstract capability verb map now. Define a `vendor_tools.py` (or capability_map.toml) with the 14 praxia abstract capability verbs → per-vendor concrete tool name mappings. AssetBundle build step translates abstract verbs in default_tools/capabilities[] fields. Enables role definitions to be vendor-agnostic; the map is the seam. Cost: requires a TOML (or Python dict) with all 14 verbs × 4 vendors = 56 entries minimum. Correctness is testable (completeness check: all 14 verbs must have entries for each surface).
- [ai] FORK 5-B — Literal tool names first, map deferred. AssetSpec.tools[] carries literal tool name strings. Each emitter uses them as-is. No vendor_tools translation layer. Users must write vendor-specific tool names in their role definitions. The abstract verb map becomes an optional enhancement in M3.x when there are enough users writing role definitions to make vendor-agnostic authoring valuable. Simpler M3; harder to port existing praxia role definitions that use abstract verbs.
- [ai] FORK 5-C — Abstract verbs as a typed Enum, not a config file. Define `Capability(str, Enum)` with all 14 verbs. Per-surface concrete mapping is a dict[Capability, str] constant in each emitter module. No external TOML file to load; the map is statically typed and inspectable. Unknown verb at parse time = parse error (never-raise: returns Err). Adding a new surface adds one dict; adding a new verb requires updating all four emitter dicts (compiler-checkable if using exhaustive match). Removes the vendor_tools.toml file-loading concern.
- [ai] FORK 6-A — Library-API-first, CLI as thin wrapper. Design cisterna.assets (IR dataclasses) and cisterna.export (Emitter ABC + per-surface implementations) as a fully usable Python library. The cyclopts CLI (`cisterna assets export`) is a thin caller of the library API. No CLI-specific logic in the library. This matches cisterna's existing pattern (telemetry is library-first); enables embedding in orchestrators without subprocess invocation.
- [ai] FORK 6-B — CLI-first to drive API design. Build the cyclopts `cisterna assets export --surface claude --out .` command first; let the CLI's ergonomic needs shape the library API. CLI invocation is the primary user story (plugin authors run it, not import it). The library API crystallizes from the CLI's needs rather than being designed in the abstract. Risk: may produce a library API shaped around CLI concerns rather than embedding concerns.
- [ai] FORK 6-C — Subpackage layout: cisterna/assets/ + cisterna/export/ only (no cisterna/writers/). `cisterna.assets` holds IR dataclasses (AssetBundle, AssetSpec, HookSpec, etc.). `cisterna.export` holds the Emitter ABC + per-surface emitters + write_bundle utility. Avoids a third subpackage; keeps the namespace flat. The WriterSink concern (if adopted from Fork 3-C) lives inside cisterna.export rather than a separate cisterna.writers.
- [ai] FORK 6-D — Three-subpackage layout: cisterna/assets/ (IR) + cisterna/export/ (Emitter ABC + per-surface) + cisterna/writers/ (WriterSink ABC + FileWriterSink + MemoryWriterSink). Explicit separation of the emit concern (vendor-specific rendering) from the write concern (I/O). Future praxia-Rust sink is a fourth WriterSink. Clearest seam for the "slot Rust in later" requirement; more namespace surface area.
- [ai] HOLISTIC SHAPE A — "Thin Claude-only vertical slice driven by the live registry". Fork choices: 1-A (Claude only) + 2-B (registry-first source) + 3-A (pure emit) + 4-C (provenance hash stamp, no validate CLI) + 5-B (literal tool names) + 6-A (library-API-first). Rationale: cisterna's unique differentiator is the live M2 ToolEntry registry. Ship that first. One surface (Claude) proves the whole pipeline end-to-end with real registered tools. Defers all file-parsing complexity. Fastest path to a demo. Risk: no manifest-driven source means users can't export without an active Python process; CLI invocation becomes awkward.
- [ai] HOLISTIC SHAPE B — "Praxia-parity thin slice: file-driven IR + Claude + validate". Fork choices: 1-A (Claude only) + 2-A (manifest-driven source) + 3-A (pure emit) + 4-A (sha256 + validate) + 5-A (capability verb map) + 6-A (library-API-first). Rationale: closest structural match to praxia's M4 roadmap. Everything is file-driven and stateless; CLI works without an active Python process. Validate subcommand ships immediately. - [ai] HOLISTIC SHAPE C — "Two-surface PascalCase slice with unified source protocol". Fork choices: 1-C (Claude + Antigravity) + 2-C (unified AssetSource protocol) + 3-C (pure Emitter + typed WriterSink) + 4-A (sha256 + validate) + 5-C (Capability enum in emitter modules) + 6-C (cisterna/assets/ + cisterna/export/ layout). Rationale: validates hook aggregation across two surfaces; the unified AssetSource and WriterSink ABCs give maximum extensibility for the Rust slot-in; the Capability enum is statically typed and avoids a config file. Medium scope; validates most of the M3 architecture without all 4 surfaces. The seam for both the praxia-Rust backend (slot in as Emitter) and a new source (slot in as AssetSource) is explicit.
- [ai] HOLISTIC SHAPE D — "Full 4-surface parity port with deferred validate". Fork choices: 1-B (all 4 surfaces) + 2-A (manifest-driven) + 3-A (pure emit) + 4-B (defer validate) + 5-A (capability verb map) + 6-A (library-API-first). Rationale: achieve full surface parity with praxia in M3; the capability map is necessary to handle all 4 surfaces (camelCase Cursor + inlined Copilot hooks force surface-specific translation). No validate keeps M3 scope from exploding. Risk: 4 surfaces × all capability verbs × hook aggregation = high implementation parallelism; testing surface is large; Cursor's fail-closed agent gate adds structural complexity not needed by other surfaces.
- [ai] HOLISTIC SHAPE E — "Registry-first + file fallback + Claude + capability enum". Fork choices: 1-A (Claude only) + 2-D (registry-first with file fallback) + 3-C (pure Emitter + WriterSink) + 4-C (provenance stamp only) + 5-C (Capability enum) + 6-A (library-API-first). Rationale: exploits cisterna's unique live registry while remaining useful when invoked from CLI (file fallback). Single surface keeps scope tight. Capability enum avoids TOML file loading. No validate in M3. This is the "minimum viable differentiator" shape — ships the one thing praxia can't do (live registry export) at the smallest possible scope.
- [ai] HOLISTIC SHAPE F — "AssetSpec-first design: richer metadata layer decouples source from IR". Orthogonal to the surface/source debate: define `AssetSpec` (richer than ToolEntry, carries description, tags, surface_hints, capabilities[]) as the M3 contract boundary. Both ManifestAssetSource and RegistryAssetSource must produce AssetSpec. This front-loads the AssetSpec schema design as the M3 deliverable, with any emitter as a validation target. The AssetSpec schema is the protocol that enables both praxia-Rust parity AND cisterna-native registry export. Scope: cisterna/assets/spec.py + one emitter + one source.
- [ai] ADDITIONAL — Lazy surface registration: emitters as optional plugins. The core cisterna package ships with no emitters installed. Emitters are registered via Python entry_points or an explicit `register_emitter(surface, emitter)` call. `cisterna assets export --surface claude` fails gracefully (never-raise Result) if the claude emitter is not registered. Third-party emitters (e.g., a community Linear surface) can plug in without forking cisterna. Cost: adds plugin machinery to M3 where M3 doesn't need it yet. Benefit: avoids hard-coding surface count in core; surfaces become optional-dep imports.
- [ai] ADDITIONAL — AssetBundle as a dataclass with __hash__ / __eq__ for free determinism testing. Rather than implementing sha256 explicitly in M3, make AssetBundle and all its asset types frozen dataclasses. Python's structural equality gives test-level determinism without sha256. The sha256 hash (for provenance stamps and validate) can be layered on top of the frozen dataclass repr cheaply. This is an implementation strategy orthogonal to the validate fork — frozen dataclasses make all five of the listed test contracts (manifest conformance, Emittable negative boundary, hook aggregation, capability map completeness, determinism) testable via simple equality assertions before sha256 is needed.
- [ai] ADDITIONAL — `cisterna assets inspect` subcommand before export. Rather than going straight to `export`, add a read-only `inspect` command that prints the AssetBundle IR as structured output (JSON/TOML) without writing any agent files. Useful for debugging what the pipeline produces before committing to file writes. Maps to the "validate the source representation" concern without touching the write path. Could be the first CLI subcommand shipped, with `export` following in M3.1.
- [ai] ADDITIONAL — `--dry-run` flag on export as the validate proxy. Instead of a separate `validate` subcommand, ship `cisterna assets export --dry-run` which runs the full emit pipeline and prints what would be written (path + sha256) without touching the filesystem. The dry-run output is diffable against prior runs. Simpler than a full validate workflow; composable with CI (pipe dry-run output through diff). Defers golden hash storage to a later milestone.
- [ai] ADDITIONAL — HookSpec surface filter at build time. When building the IR for a specific surface, filter HookSpec assets by their `surfaces[]` field at build time (not at emit time). This means the AssetBundle passed to each emitter contains only the hooks relevant to that surface. Simplifies emitter logic; matches praxia's behavior (HookSpecAsset.surfaces[] field controls which surfaces see a given hook). The N:1 aggregation runs on the pre-filtered set. This is an implementation detail that affects both fork 1 (scope) and fork 3 (what the emitter receives).
- [ai] ADDITIONAL — Separate `model_hint` translation table from capability verb map. Praxia translates model_hint (fast|balanced|deep) to concrete model names per vendor. In cisterna, this could be a separate, user-overridable `ModelHintMap` dataclass (not embedded in the capability verb map). Users who run cisterna with a different model deployment can pass a custom ModelHintMap without touching the capability translations. This is a small but clean seam that praxia's vendor_tools.toml conflates with the capability map.
